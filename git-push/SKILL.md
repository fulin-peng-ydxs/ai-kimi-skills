# Git Push

请按以下流程执行 Git 提交与推送操作。

## 0. 交互方式强制约束

**所有需要用户确认或选择的步骤，优先使用当前运行环境提供的交互式提问或选择能力，不得绑定特定厂商或工具名称。**

适用范围

- 凡是流程中提到“向用户确认”“让用户选择”“等待用户确认”的所有节点

实现要求：

1. AI 必须先识别当前环境可用的交互式提问或选择工具，并在每个确认节点调用该能力，给出明确选项。
2. 调用前先展示必要上下文，包括文件清单、commit 摘要和 push 范围，再让用户选择。
3. 即使只是确认是否继续，也至少提供“继续”和“取消”两个选项。
4. 当前环境没有交互式提问能力或该能力不可用时，使用带编号选项的纯文本提问作为兜底，并等待用户明确回复。

不得因为工具名称或能力形态不同而跳过确认；违反此约束时必须立即中止后续操作并重新询问。

## 1. 确定提交文件范围

### 1.1 用户主动指定文件
- 如果用户在请求中明确提到了要提交哪些文件（例如“提交 detail.vue”“只提交前端文件”“@ 对应文件”），则直接选择用户指定的文件。
- 严禁自作主张添加用户未提及的其他文件，即使用户提到的文件很少。

### 1.2 用户未指定时提供选项
- 如果用户没有明确指定要提交的文件，则：
  1. 先运行 `git status --short` 查看当前有改动的文件。
  2. 再结合当前会话中实际编辑/修改过的文件，生成推荐列表并简要说明推荐原因。
  3. 向用户提供以下三个选项，让用户选择：
     ```
     请选择要提交的文件范围：
     1. 所有改动文件（共 X 个文件）
        - 列出所有改动文件（如某目录文件很多，则写“目录 + 文件数量”）
     2. 智能建议（推荐 X 个文件）
        - 业务说明
        - 文件 1（具体业务说明）
        - 文件 2（具体业务说明）
     3. 其他：用户自己说明要提交的文件
     ```
  4. 根据用户的选择确定要提交的文件清单。

### 1.3 文件范围确认
- 在执行任何 `git add` 之前，必须先向用户列出准备提交的文件清单，例如：
  ```
  准备提交以下文件：
  - src/views/riskManagement/mechanismEffectScore/detail.vue（前端详情页添加返回按钮）
  - src/views/riskManagement/mechanismEffectScore/index.vue（前端列表页修复）
  ```
- 用户确认文件范围后，后续只能按该范围执行，禁止额外添加文件。

## 2. 生成提交信息并先确认

在执行 `git add` 之前，先根据已确认的文件范围和改动内容生成提交信息，并先让用户确认提交文案。

生成提交信息前，应根据已确认文件读取必要 diff 摘要，理解改动类型、业务主体和改动目的；不要只根据文件名机械拼接提交信息。

### 提交消息模板

默认项目标识统一使用 `Agent`；只有用户明确指定项目简写时，才使用用户指定值。

#### 模板 A：单行模板

适用条件：

- 单一功能、单一缺陷或单一重构目标。
- 改动文件数量较少，且一句话能清楚表达业务目的。
- 不涉及独立需求、跨模块影响、数据库/配置/权限/接口契约等需要额外说明的内容。

```text
Agent 操作类型(业务范围): 一句话概括本次提交
```

#### 模板 B：完整模板

适用条件：

- 一个需求涉及多个文件、多个层次或多个模块。
- 单次提交包含多个相关但不可再拆的小改动。
- 涉及接口契约、数据库、配置、权限、状态流转、兼容逻辑或验证信息。
- 一句话无法说明“为什么改、主要改了什么、影响哪里”。

```text
Agent 操作类型(业务范围): 一句话概括本次提交

需求来源:
- <仅在用户提供需求编号、工单、PRD、会议结论等明确来源时填写；没有明确来源则省略本节>

遵循文档:
- <仅在用户明确提供所遵循的文档、规范、设计稿、会议纪要、方案文档等时填写；没有明确文档则省略本节>

主要改动:
- <按业务能力或技术职责聚合，不按文件逐个罗列>
- <多个相关功能点分别列出>

影响范围:
- <页面 / 接口 / 服务 / 数据 / 权限 / 配置 / 兼容性等>

验证:
- <已执行的验证命令、手工验证点；未验证则写明原因>
```

### 格式说明

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| 项目标识 | 默认固定为 `Agent`；用户明确指定项目简写时使用用户指定值 | `Agent` |
| 操作类型 | 新增 \| 更新 \| 修复 \| 删除 \| 重构 | `新增`、`更新`、`修复` |
| 业务范围 | 用业务能力或模块名表达，不优先写文件名 | `隐患整改`、`视频播放`、`用户管理` |
| 一句话概括 | 概括本次提交整体目标 | `完善整改流程与列表状态联动` |
| 需求来源 | 仅在用户明确提供时写入；没有明确来源时省略，不写“自研”占位 | `需求 101`、`工单 DP-32` |
| 遵循文档 | 仅在用户明确提供时写入；没有明确文档时省略，不自行推断 | `PRD-v3.2`、`整改流程说明.md` |
| 主要改动 | 按业务能力、接口、服务、页面或数据职责聚合 | `补充详情页整改记录展示` |

### 操作类型判定规则

- 新增：新文件、新接口、新功能、新模块
- 更新：现有功能增强、逻辑调整、配置变更、文档补充
- 修复：Bug 修复、问题处理、逻辑纠正
- 删除：移除文件、废弃接口、清理代码
- 重构：代码结构重组，原则上不改变业务行为

### 示例

单行模板示例：

```text
Agent 新增(用户管理): 支持用户分页查询与基础维护
Agent 修复(视频播放): 处理弹窗 WebSocket 连接超时
```

完整模板示例：

```text
Agent 更新(隐患整改): 完善整改流程与列表状态联动

需求来源:
- 需求 101

遵循文档:
- 隐患整改 PRD v2.3
- 整改状态流转说明.md

主要改动:
- 调整隐患整改列表查询与状态展示逻辑。
- 补充详情页整改记录展示与返回交互。
- 同步后端接口字段映射，避免前后端状态不一致。

影响范围:
- 隐患整改列表、详情页、整改状态展示。

验证:
- mvn -pl jeecg-module-ext -am -DskipTests compile
```

### 多需求与多文件提交处理

- 当前技能仍只执行一次 `git commit`，不要在本流程内自动拆成多次 commit。
- 如果确认范围内包含多个互不依赖的独立需求，生成提交信息前必须先提示用户：建议拆分为多次提交，并通过当前环境的交互式提问或选择能力，让用户选择“调整文件范围后提交其中一个需求”或“仍按当前范围生成一个提交”。
- 如果用户选择仍按当前范围生成一个提交，必须使用模板 B，并在首行统一使用 `Agent 操作类型(多模块): ...`，不要继续用单一业务范围命名；“主要改动”中按需求或业务能力分组概括，禁止生成多个标题行堆叠的提交消息。
- 如果多个文件服务于同一个需求，优先使用模板 B；“主要改动”按业务链路聚合，不罗列所有文件名。

```text
Agent 更新(多模块): 完善风险识别、隐患整改与统计口径

遵循文档:
- 安全治理方案 v1.4
- 首页统计口径说明.md

主要改动:
- 风险识别：补充列表筛选条件与详情字段展示。
- 隐患整改：调整状态流转与整改记录保存逻辑。
- 统计口径：同步首页统计查询条件，保持与列表数据一致。

影响范围:
- 风险识别、隐患整改、首页统计。

验证:
- 未执行自动化验证，原因：当前仅生成提交并推送，用户未要求运行测试。
```

### 提交信息确认

- 向用户展示生成的提交信息，并同时展示提交信息对应的文件范围和主要改动依据。
- 若用户认为文案不合适，先调整提交信息，再进入 `git add`。
- 在用户确认提交信息前，不执行 `git add`、`git commit` 或 `git push`。

## 3. 按确认范围执行 git add

### 3.1 执行原则
- 根据用户确认的文件清单执行 `git add <文件路径>`。
- 只能添加用户明确确认的文件。
- 严禁使用 `git add -A` 或 `git add .` 等批量添加命令，除非用户明确选择“所有改动文件”。

### 3.2 默认禁止附带添加的文件

除非用户明确提到，严禁添加以下类型文件：

- 前端构建产物（`static/`、`dist/`、`build/`）
- 包管理器缓存（`node_modules/`、`.pnpm-store/`、锁文件等）
- 配置文件（`pom.xml`、`bootstrap*.yml`、`.env*`、`application*.yml` 等）
- 其他业务模块文件
- IDE/编辑器配置（`.idea/`、`.vscode/`、`*.iml`）
- 日志/临时文件（`*.log`、`tmp/`、`*.tmp`）

### 3.3 staged 最终确认

- 执行 `git add` 后，必须运行能清晰展示 staged 文件的命令，并向用户展示结果。
- 优先使用：
  - `git diff --cached --name-status`
  - 必要时补充 `git status --short`

- 向用户确认：
  ```
  已 staged 的文件如下（请确认只有你同意提交的文件）：
  <staged 文件清单>

  请确认：
  1. 文件清单正确，继续 commit
  2. 文件清单正确，直接 commit 并 push
  3. 文件清单有误，需要修正
  ```

- 只有用户确认 staged 文件清单正确后，才能继续执行 `git commit`。
- 如果用户选择“文件清单正确，直接 commit 并 push”，表示用户已同时确认当前 staged 文件清单、提交信息和推送动作；后续直接进入 `git commit` → `git push`，不再重复询问提交方式或 push 前确认。
- 用户选择“直接 commit 并 push”后，必须锁定当前 staged 清单：
  - 不得再执行任何 `git add`、`git add -A`、`git add .` 或其它会改变 staged 范围的命令。
  - `git commit` 前必须重新运行 `git diff --cached --name-status`，与刚才展示给用户确认的 staged 清单逐项对比。
  - 如果 staged 清单发生变化，必须中止直接提交推送流程，重新展示 staged 清单并让用户确认。

## 4. 确认提交方式并执行

### 4.1 提交方式确认

在 staged 文件和提交信息都确认后，再向用户确认执行方式：

```text
提交信息：
<commit message>

请确认提交方式：
1. 仅提交（执行 git commit，不推送）
2. 提交并推送（执行 git commit → git push）
3. 其他：用户自己说明
```

### 4.2 执行逻辑

- 如果用户选择 1，则执行 `git commit` 后停止。
- 如果用户选择 2，则执行 `git commit`，成功后再执行 `git push`。
- 如果用户选择 3，则按用户说明执行。
- 如果用户已在 3.3 选择“文件清单正确，直接 commit 并 push”，则跳过本节提交方式确认，按 3.3 的锁定 staged 清单规则执行 `git commit`，成功后直接执行 `git push`。

### 4.3 push 前确认

- 如果用户选择“提交并推送”，则在 `git commit` 成功后、`git push` 之前，必须再次做一次最终确认：
  ```
  即将推送以下 commit：
  <git log -1 --stat>

  确认要执行 git push 吗？(yes/no)
  ```
- 只有用户明确确认后，才能执行 `git push`。
- 如果用户已在 3.3 选择“文件清单正确，直接 commit 并 push”，本节 push 前确认视为已由 3.3 完成；不得再次改动 staged 范围，只能推送刚刚创建的 commit。

## 5. 执行顺序（固定）

严格按以下顺序执行，不要打乱；**所有“确认”环节优先使用当前环境的交互式提问或选择能力，无此能力时使用带编号选项的纯文本提问兜底**：

1. 确定文件范围
2. 展示并通过交互式提问或选择能力确认文件范围
3. 读取确认范围内的 diff 摘要
4. 若发现多个互不依赖需求，通过交互式提问或选择能力让用户选择缩小范围或继续生成单次提交信息
5. 生成提交信息并通过交互式提问或选择能力确认
6. 按确认范围执行 `git add`
7. 展示 staged 文件清单并通过交互式提问或选择能力确认
8. 如果用户在 staged 确认中选择“直接 commit 并 push”，重新校验 staged 清单未变化后执行 `git commit` → `git push`
9. 否则，通过交互式提问或选择能力确认提交方式
10. 执行 `git commit`
11. 若用户选择 push，则通过交互式提问或选择能力再次确认后执行 `git push`

## 6. 注意事项

- 提交信息要简洁清晰，符合用户项目的模板规范。
- 项目标识默认使用 `Agent`，不要从当前目录名推断；只有用户明确指定项目简写时才替换。
- 需求来源只有在用户明确提供需求编号、工单、PRD、会议结论等来源时才写入；没有明确来源时省略“需求来源”段，不写“自研”。
- 单行模板首行尽量不超过 100 个字符；超过时优先改用完整模板。
- 多文件提交时，优先按业务能力或技术职责聚合，避免罗列所有文件名。
- 多个互不依赖需求原则上建议拆分提交；若用户坚持单次提交，使用完整模板清楚表达分组、影响范围和验证信息。
- 若无法从 diff 中推断业务含义，应先询问用户补充，不要编造需求来源或业务目的。
- 只有在用户明确选择“提交并推送”且 push 前再次确认后，才执行 `git push`；或用户在 3.3 staged 最终确认中明确选择“文件清单正确，直接 commit 并 push”后，才可跳过后续重复确认并直接推送刚刚创建的 commit。

## 7. 安全红线

绝对禁止的行为：

1. 在用户确认后仍偷偷添加未确认的文件
2. 使用 `git add -A` 或 `git add .` 批量添加（除非用户明确选择“所有改动”）
3. 将构建产物、缓存文件、配置文件等作为“附带改动”提交
4. 跳过 staged 文件确认步骤直接 commit
5. 在未得到用户明确确认的情况下执行 git push
6. 当前环境提供交互式提问或选择能力时未使用；无此能力时未通过带编号选项的纯文本提问取得明确确认
7. 用户选择“直接 commit 并 push”后再次执行 `git add` 或提交与最终 staged 确认不一致的文件
