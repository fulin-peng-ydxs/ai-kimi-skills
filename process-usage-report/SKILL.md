# Process Usage Report

这个技能用于查看当前电脑的整体进程使用情况，并输出一份适合排查卡顿问题的报告。

目标不是直接结束进程，而是先回答下面这些问题：

1. 当前有多少进程正在运行。
2. 哪些进程占用 CPU、内存、IO 最多。
3. 这些进程大致属于什么类型。
4. 当前卡顿更可能是由哪类进程引起。
5. 后续应该优先处理哪些进程。

## 执行流程

1. 先采集当前机器的进程快照。
2. 统计总体进程数量和基础分布。
3. 找出 CPU、内存、IO 资源占用最高的进程。
4. 对进程做分类统计，帮助识别是哪一类程序在持续占用资源。
5. 输出结构化报告，明确最可能导致卡顿的嫌疑进程和处理建议。

## 采集重点

优先关注：

1. 进程总数
2. 用户进程和系统进程数量
3. CPU 占用最高的进程
4. 内存占用最高的进程
5. IO 活动明显的进程或整体 IO 压力
6. 浏览器、IDE、Java、Node、Docker、数据库、聊天工具、同步工具、系统服务等分类分布

## 分析要求

1. 不只列出单个高占用进程，还要判断是否存在某一类进程整体占用过高。
2. 不只看瞬时 CPU，还要结合内存、线程数、子进程数量、启动器/宿主进程等信息判断。
3. 若某类程序开了很多子进程，要尽量合并说明，不要只罗列碎片化进程名。
4. 若 IO 无法可靠定位到具体进程，要明确说明限制，不要伪造结论。
5. 若发现明显异常，例如：
   - Java/Node 进程长期高占用
   - 浏览器标签页或扩展导致大量辅助进程
   - Docker/虚拟化进程持续占资源
   - 同步盘、索引、杀毒或系统服务大量扫描
   - 某类工具反复拉起子进程
   都要单独点出来。

## 输出要求

输出时至少包含：

1. 总体概览
   - 当前总进程数
   - 用户进程与系统进程的大致分布
   - 当前整体 CPU / 内存 / IO 压力判断

2. 高占用进程
   - CPU Top
   - 内存 Top
   - IO Top

3. 进程分类统计
   - 各主要类别的进程数量
   - 各主要类别的大致资源占用情况

4. 卡顿嫌疑分析
   - 最可疑的 1 到 3 类进程或具体进程
   - 为什么怀疑它们

5. 处理建议
   - 建议优先关闭、重启、排查或观察哪些进程
   - 如果需要进一步精准处理，再建议用户确认是否执行具体操作

## 执行约束

1. 默认只做查看、统计和分析，不主动结束进程。
2. 需要结束、重启、kill 某个进程时，必须先获得用户明确同意。
3. 结论基于当前时刻快照，若资源波动明显，要明确说明这是瞬时观察结果。
4. 若某些指标当前环境下拿不到，就明确说明限制，不要编造数据。
