# Disk Usage Report

这个技能用于查看当前电脑的磁盘空间使用情况，并输出一份适合排查空间不足、指导清理的报告。

目标不是直接删文件，而是先回答下面这些问题：

1. 当前磁盘总体用了多少、还剩多少。
2. 哪些目录占用的空间最多。
3. 这些目录里大致存的是什么东西。
4. 哪些目录适合清理、哪些绝对不能动。
5. 后续应该优先清理哪些内容。

## 执行流程

1. 先采集当前机器的整体磁盘使用快照。
2. 再对目标目录（默认用户主目录或用户指定目录）做分层空间统计。
3. 识别占用最高的目录，并分析其内容类型。
4. 判断哪些内容可安全清理，哪些需要谨慎处理或保留。
5. 输出结构化报告，明确清理建议和操作边界。

## 采集重点

优先关注：

1. 磁盘总体容量、已用空间、可用空间、使用比例
2. 目标目录下占用 Top 的子目录
3. 常见“空间大户”目录：
   - 下载目录（`~/Downloads`）
   - 应用缓存与日志（`~/Library/Caches`、`~/Library/Logs`、`.cache/`）
   - 开发与构建产物（`node_modules/`、`target/`、`build/`、`dist/`、`.gradle/`）
   - 容器与虚拟化（Docker 镜像与数据卷、虚拟机镜像）
   - 聊天与同步工具缓存（微信、QQ、钉钉、OneDrive、iCloud、百度网盘等）
   - IDE 与编辑器索引（`.idea/`、`.vscode/`、`.claude/`、`.cursor/`、`.continuum/` 等）
   - 包管理器缓存（`~/.npm/`、`~/.pnpm-store/`、`~/.m2/`、`~/.conda/`、`~/.pip/`）
4. 大文件或重复文件的明显线索（单个大文件、异常膨胀的日志/数据库）
5. 系统级目录只查看、不触碰（如 `/System`、`/Library`、`/usr`、`/bin` 等）

## 分析要求

1. 不只列出单个大目录，还要判断是否存在某一类目录整体膨胀。
2. 要结合目录名、常见用途和实际内容，推断里面“存的是什么”。
3. 对可清理目录给出具体理由；对不可清理目录要明确标注风险。
4. 若某些目录需要进一步下钻才能确定内容，要说明“需要再深入一层查看”。
5. 若发现明显异常，例如：
   - 某个应用缓存目录异常膨胀
   - 日志文件单个体积过大或长期未轮转
   - 旧的构建产物、安装包、临时文件堆积
   - Docker/虚拟机镜像长期未清理
   - 同步盘重复缓存多份副本
   都要单独点出来。

## 输出要求

输出时至少包含：

1. 总体概览
   - 磁盘总容量、已用、可用、使用比例
   - 若存在多个卷 / 分区，分别列出
   - 当前整体磁盘压力判断（宽松 / 紧张 / 危急）

2. 空间占用 Top 目录
   - 占用最高的 10 个目录（或按用户指定范围）
   - 每个目录的占用大小
   - 该目录大致存放的内容类型

3. 目录分类统计
   - 缓存与日志
   - 开发与构建产物
   - 容器与虚拟化
   - 聊天与同步工具
   - 下载与文档
   - 其他大目录
   - 各类别的大致占用合计

4. 可清理目标分析
   - 推荐优先清理的 1 到 5 个目标
   - 每个目标预计可释放空间
   - 清理风险等级（安全 / 谨慎 / 不建议自行处理）

5. 清理建议
   - 具体可执行的清理动作（如清空废纸篓、删除旧安装包、清理 Docker 镜像、清空应用缓存等）
   - 哪些动作需要用户确认后再执行
   - 哪些动作建议用户自行处理（如聊天记录备份后删除）

## 清理确认流程

只要用户表达中包含“清理”意图，无论形式是“清理第 xxx 项”“清理指定目录”“清理 xxx 内容”还是其他文字说明，都必须先将要清理的内容完整列出，并通过 `AskUserQuestion` 工具弹出确认窗口，获得用户明确同意后才能执行删除。

### 1. 解析用户清理指令

用户可能通过以下方式表达清理意图：

- **按编号**：例如“清理第 1、3 项”“清理前 5 个”
- **按目录**：例如“清理 ~/Downloads”“清理 Docker 镜像”“清理 Xcode 缓存”
- **按描述**：例如“清理日志”“清理旧的安装包”“清理微信缓存”
- **混合方式**：例如“清理第 2 项和第 4 项，再加 ~/Library/Logs”

无论哪种方式，都必须先明确映射到具体的文件或目录清单。

### 2. 列出待清理内容

在执行任何删除操作前，必须先向用户展示：

```text
根据你的指令，准备清理以下内容：

1. /path/to/目录A（预计释放 X GB）— 存放内容说明
2. /path/to/文件B（预计释放 Y MB）— 存放内容说明
3. ...

请确认是否执行清理：
1. 确认清理
2. 取消，只保留分析报告
3. 调整清理范围（用户重新说明）
```

展示要求：

1. 列出每个待清理项的完整路径。
2. 说明每个待清理项的大致内容类型。
3. 给出每个待清理项的预计释放空间（若无法精确计算，可给出估算范围）。
4. 对风险较高的项单独标注“谨慎”或“涉及应用数据，建议先备份”。

### 3. 使用 AskUserQuestion 强制确认

**所有清理确认必须使用 `AskUserQuestion` 工具弹出选项窗口，禁止使用纯文本提问让用户手动输入回复。**

实现要求：

1. 每个清理确认节点都必须显式调用 `AskUserQuestion`。
2. 在调用 `AskUserQuestion` 前，先用普通文本展示待清理清单。
3. 确认选项至少包含：
   - 确认清理
   - 取消清理
   - 调整范围
4. 用户未明确选择“确认清理”之前，不得执行任何删除、清空、`rm`、`docker system prune`、`brew cleanup` 等操作。

### 4. 执行清理

只有在用户通过 `AskUserQuestion` 明确选择“确认清理”后，才能执行删除。

执行原则：

1. 严格按确认清单执行，不得额外添加未确认项。
2. 优先使用安全的清理方式（如移到废纸篓、应用自带的清理功能），避免直接使用 `rm -rf`。
3. 若某项确实需要使用命令行删除，必须先说明将要执行的命令，并在确认窗口中再次明确。
4. 清理完成后，重新采集磁盘快照，向用户展示释放的空间。

## 执行约束

1. 默认只做查看、统计和分析，不主动删除任何文件或目录。
2. 需要清理、删除、清空、卸载等操作时，必须先完成上述“清理确认流程”。
3. 严禁删除系统目录和关键应用数据；即使空间紧张，也不建议触碰 `/System`、`/Library`、系统根目录等。
4. 对用户主目录下的敏感内容（如聊天记录、工作文档、密钥配置）要谨慎建议，避免误删。
5. 结论基于当前时刻快照，若目录内容动态变化，要明确说明这是瞬时观察结果。
6. 若某些指标或目录内容当前环境下拿不到，就明确说明限制，不要编造数据。
