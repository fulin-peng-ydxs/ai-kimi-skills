# Git Commit Msg

根据当前 Git 工作区的改动，生成符合项目规范的提交消息。

## 0. 交互方式强制约束

**所有需要用户确认或选择的步骤，必须使用 `AskUserQuestion` 工具弹出选项窗口，禁止使用纯文本提问让用户手动输入回复。**

适用范围：

- 凡是流程中提到“向用户确认”“让用户选择”“等待用户确认”的所有节点

实现要求：

1. 每个确认节点都必须显式调用 `AskUserQuestion`，给出明确选项。
2. 必要的上下文（文件清单、范围说明等）在调用前先用普通文本展示，再用 `AskUserQuestion` 让用户点选。
3. 即使是简单的“是否继续”，也必须用弹窗（至少 2 个选项：继续 / 取消）。

## 1. 确定生成提交消息的文件范围

### 1.1 用户主动指定文件

- 如果用户在请求中明确提到了要基于哪些文件生成提交消息（例如“给 detail.vue 生成提交消息”“只看前端文件”“@ 对应文件”），则直接选择用户指定的文件。
- 严禁自作主张加入用户未提及的其他文件。

### 1.2 用户未指定时提供选项

- 如果用户没有明确指定文件范围，则：
  1. 运行 `git status --short` 查看当前有改动的文件。
  2. 结合当前会话中实际编辑/修改过的文件，生成推荐列表并简要说明推荐原因。
  3. 向用户提供以下三个选项：
     ```text
     请选择生成提交消息的文件范围：
     1. 所有改动文件（共 X 个文件）
        - 列出所有改动文件（如某目录文件很多，则写“目录 + 文件数量”）
     2. 智能建议（推荐 X 个文件）
        - 业务说明
        - 文件 1（具体业务说明）
        - 文件 2（具体业务说明）
     3. 其他：用户自己说明要生成提交消息的文件
     ```
  4. 根据用户选择确定文件清单。

### 1.3 文件范围确认

- 在生成提交消息前，必须先向用户列出准备分析的文件清单，例如：
  ```text
  准备基于以下文件生成提交消息：
  - src/views/riskManagement/mechanismEffectScore/detail.vue（前端详情页添加返回按钮）
  - src/views/riskManagement/mechanismEffectScore/index.vue（前端列表页修复）
  ```
- 用户确认文件范围后，后续只能基于该范围生成提交消息。

## 2. 分析改动内容

1. 对已确认文件运行必要的 diff 查看命令，例如 `git diff -- <file>` 或 `git diff --cached -- <file>`。
2. 判断改动类型、业务主体、需求来源和具体内容。
3. 不要只根据文件名机械生成提交消息。
4. 若无法从 diff 中判断业务含义，应先询问用户补充，不要编造需求来源或业务目的。

## 3. 生成提交消息

### 提交消息模板

默认项目标识统一使用 `Agent`；只有用户明确指定项目简写时，才使用用户指定值。

#### 模板 A：单行模板

适用条件：

- 单一功能、单一缺陷或单一重构目标。
- 改动文件数量较少，且一句话能清楚表达业务目的。
- 不涉及独立需求、跨模块影响、数据库/配置/权限/接口契约等需要额外说明的内容。

```text
Agent 操作类型(业务范围): 一句话概括本次提交
```

#### 模板 B：完整模板

适用条件：

- 一个需求涉及多个文件、多个层次或多个模块。
- 单次提交包含多个相关但不可再拆的小改动。
- 涉及接口契约、数据库、配置、权限、状态流转、兼容逻辑或验证信息。
- 一句话无法说明“为什么改、主要改了什么、影响哪里”。

```text
Agent 操作类型(业务范围): 一句话概括本次提交

需求来源:
- <仅在用户提供需求编号、工单、PRD、会议结论等明确来源时填写；没有明确来源则省略本节>

遵循文档:
- <仅在用户明确提供所遵循的文档、规范、设计稿、会议纪要、方案文档等时填写；没有明确文档则省略本节>

主要改动:
- <按业务能力或技术职责聚合，不按文件逐个罗列>
- <多个相关功能点分别列出>

影响范围:
- <页面 / 接口 / 服务 / 数据 / 权限 / 配置 / 兼容性等>

验证:
- <已执行的验证命令、手工验证点；未验证则写明原因>
```

### 格式说明

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| 项目标识 | 默认固定为 `Agent`；用户明确指定项目简写时使用用户指定值 | `Agent` |
| 操作类型 | 新增 \| 更新 \| 修复 \| 删除 \| 重构 | `新增`、`更新`、`修复` |
| 业务范围 | 用业务能力或模块名表达，不优先写文件名 | `隐患整改`、`视频播放`、`用户管理` |
| 一句话概括 | 概括本次提交整体目标 | `完善整改流程与列表状态联动` |
| 需求来源 | 仅在用户明确提供时写入；没有明确来源时省略，不写“自研”占位 | `需求 101`、`工单 DP-32` |
| 遵循文档 | 仅在用户明确提供时写入；没有明确文档时省略，不自行推断 | `PRD-v3.2`、`整改流程说明.md` |
| 主要改动 | 按业务能力、接口、服务、页面或数据职责聚合 | `补充详情页整改记录展示` |

### 操作类型判定规则

- 新增：新文件、新接口、新功能、新模块
- 更新：现有功能增强、逻辑调整、配置变更、文档补充
- 修复：Bug 修复、问题处理、逻辑纠正
- 删除：移除文件、废弃接口、清理代码
- 重构：代码结构重组，原则上不改变业务行为

### 示例

单行模板示例：

```text
Agent 新增(用户管理): 支持用户分页查询与基础维护
Agent 修复(视频播放): 处理弹窗 WebSocket 连接超时
```

完整模板示例：

```text
Agent 更新(隐患整改): 完善整改流程与列表状态联动

需求来源:
- 需求 101

遵循文档:
- 隐患整改 PRD v2.3
- 整改状态流转说明.md

主要改动:
- 调整隐患整改列表查询与状态展示逻辑。
- 补充详情页整改记录展示与返回交互。
- 同步后端接口字段映射，避免前后端状态不一致。

影响范围:
- 隐患整改列表、详情页、整改状态展示。

验证:
- mvn -pl jeecg-module-ext -am -DskipTests compile
```

### 多需求与多文件提交处理

- 当前技能只生成一次提交消息，不执行 `git add`、`git commit` 或 `git push`，也不要在本流程内自动拆成多条提交消息分别处理。
- 如果确认范围内包含多个互不依赖的独立需求，生成提交消息前必须先提示用户：建议拆分为多次提交，并通过 `AskUserQuestion` 让用户选择“调整文件范围后只为其中一个需求生成提交消息”或“仍按当前范围生成一个提交消息”。
- 如果用户选择仍按当前范围生成一个提交消息，必须使用模板 B，并在首行统一使用 `Agent 操作类型(多模块): ...`，不要继续用单一业务范围命名；“主要改动”中按需求或业务能力分组概括，禁止生成多个标题行堆叠的提交消息。
- 如果多个文件服务于同一个需求，优先使用模板 B；“主要改动”按业务链路聚合，不罗列所有文件名。

```text
Agent 更新(多模块): 完善风险识别、隐患整改与统计口径

遵循文档:
- 安全治理方案 v1.4
- 首页统计口径说明.md

主要改动:
- 风险识别：补充列表筛选条件与详情字段展示。
- 隐患整改：调整状态流转与整改记录保存逻辑。
- 统计口径：同步首页统计查询条件，保持与列表数据一致。

影响范围:
- 风险识别、隐患整改、首页统计。

验证:
- 未执行自动化验证，原因：当前仅生成提交消息，未运行验证命令。
```

## 4. 输出结果

生成提交消息后，直接输出最终提交消息文案，不再增加提交消息确认节点，不再通过 `AskUserQuestion` 二次确认提交消息。

```text
提交消息如下：

{生成的提交消息}
```

不要输出 `git commit -m` 命令，除非用户明确要求。

## 5. 执行顺序（固定）

严格按以下顺序执行，不要打乱；**文件范围确认必须使用 `AskUserQuestion` 弹窗，禁止纯文本提问**：

1. 确定文件范围
2. 展示并通过 `AskUserQuestion` 确认文件范围
3. 读取确认范围内的 diff 摘要
4. 若发现多个互不依赖需求，通过 `AskUserQuestion` 让用户选择缩小范围或继续生成单次提交消息
5. 生成提交消息
6. 直接输出最终提交消息

## 6. 注意事项

- 提交消息要简洁清晰，符合用户项目的模板规范。
- 项目标识默认使用 `Agent`，不要从当前目录名推断；只有用户明确指定项目简写时才替换。
- 需求来源只有在用户明确提供需求编号、工单、PRD、会议结论等来源时才写入；没有明确来源时省略“需求来源”段，不写“自研”。
- 单行模板首行尽量不超过 100 个字符；超过时优先改用完整模板。
- 多文件提交时，优先按业务能力或技术职责聚合，避免罗列所有文件名。
- 多个互不依赖需求原则上建议拆分提交消息范围；若用户坚持单次提交消息，使用完整模板清楚表达分组、影响范围和验证信息。
- 只生成提交消息，不执行任何 `git add`、`git commit` 或 `git push` 操作。

## 7. 安全红线

绝对禁止的行为：

1. 在用户确认文件范围后，额外分析未确认的文件
2. 使用纯文本提问替代 `AskUserQuestion` 弹窗
3. 编造需求编号、需求来源或业务目的
4. 生成提交消息后额外增加提交消息确认节点，导致无法直接输出
5. 执行任何 `git add`、`git commit` 或 `git push` 操作
